# 테스트 코드 작성 공통 컨벤션

이 문서는 현재 `3-team-Tasteam-be` 테스트 코드 전반에 공통으로 적용하는 기준과,
아직 전면 적용되지는 않았지만 계속 맞춰 가는 권장 기준을 구분해 정리한다.

## 1. 현재 공통 표준

- 테스트는 구현이 아니라 **행위, 결과, 상태 변화**를 검증한다.
- 실패 메시지는 디버깅 보조가 아니라 **원인 설명** 역할을 해야 한다.
- Given / When / Then 구조를 유지한다.
  - `when`은 가능하면 검증 대상 액션 1회만 둔다.
  - `then`에는 검증 외 준비 로직을 넣지 않는다.
- AssertJ를 기본으로 사용한다.
- 랜덤 값, 현재 시각 직접 호출, 로그 출력 검증, private 메서드 직접 테스트, 테스트 간 상태 공유는 지양한다.

## 2. 현재 널리 적용된 관례

### 2.1 `@DisplayName`

- 메서드 `@DisplayName`은 비즈니스 문장으로 작성한다.
- 클래스 `@DisplayName`은 아래 형식이 많이 사용된다.
  - `[유닛](도메인) Xxx 단위 테스트`
  - `[통합](도메인) Xxx 통합 테스트`
  - `[성능] Xxx 동시성 테스트`

예시:

- `@DisplayName("유효한 리프레시 토큰이면 새로운 액세스 토큰이 발급된다")`
- `@DisplayName("이미지와 함께 리뷰를 생성하면 이미지가 ACTIVE 상태로 변경된다")`

### 2.2 `@Nested`

- 유즈케이스나 기능 단위로 묶을 때 사용한다.
- 보통 2 depth 이내로 유지한다.

### 2.3 테스트 메서드 이름

- 서술형 camelCase를 사용한다.
- 구현 세부보다 행위/조건/결과가 드러나도록 쓴다.

예시:

- `createReviewWithImages`
- `getNotifications_orderedByIdDesc`
- `refreshTokens_withSameTokenConcurrently`

## 3. 권장 기준

아래는 현재 문서 기준의 권장안이며, 아직 모든 테스트 클래스에 강제되지는 않는다.

### 3.1 클래스 `@DisplayName` 권장 패턴

- 유닛: `[유닛](도메인) 클래스명 단위 테스트`
- 통합: `[통합](도메인) 클래스명 통합 테스트`
- 성능: `[성능] 클래스명 동시성 테스트`

### 3.2 한 테스트가 검증해야 하는 범위

- 테스트 1개 = 개념 1개를 권장한다.
- 성공/실패/상태 변화를 한 테스트에 과도하게 섞지 않는다.

### 3.3 테스트 데이터 준비

- 생성자 직접 호출보다 Fixture 또는 의미 있는 팩토리 메서드를 우선한다.
- 단, 도메인 테스트에서 SUT 자체는 실제 팩토리/생성자를 직접 호출하는 편이 좋다.

## 4. 리뷰 체크리스트

- 테스트 이름만 읽어도 시나리오를 이해할 수 있는가
- `when` 액션이 하나로 보이는가
- 핵심 결과 또는 상태 변화가 드러나는가
- 구현 세부 `verify`에 과도하게 결합되지 않았는가
- Fixture나 Fake가 테스트 의도를 가리고 있지 않은가

## 5. 현재 갭

- 일부 클래스 `@DisplayName`은 권장 포맷을 따르지 않는다.
  - 예: `NotificationServiceIntegrationTest`
- 일부 `@PerformanceTest` 클래스는 이름/표시는 통합 테스트처럼 보인다.
  - 예: `SearchServiceConcurrencyIntegrationTest`
- 이 문서는 권장 기준을 명시하지만, 이번 문서화 작업은 테스트 코드 자체를 일괄 수정하지 않는다.
