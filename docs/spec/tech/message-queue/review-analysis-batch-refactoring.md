## 1. 현행 배치 시스템의 구조적 특징

현재 리뷰 분석 시스템은 음식점 단위로 전체 데이터를 재분석하는 **시간 기반 배치 처리 구조**로 구성되어 있다.

배치 실행 시 다음과 같은 흐름이 반복된다:

1. 분석 대상 음식점을 넓게 조회
2. 음식점별 `AiJob` 생성
3. Executor 기반 병렬 처리
4. 작업 상태(`PENDING → RUNNING → COMPLETED`)를 RDB에서 관리

이 구조의 핵심 특징은 다음과 같다:

- **전체 재계산 중심 처리**: 특정 변경이 아닌 전체 음식점을 주기적으로 재분석
- **시간 기반 트리거**: 데이터 유입과 무관하게 배치 시점에 실행
- **RDB 기반 오케스트레이션**: 작업 생성, 상태 관리, 실행 흐름을 모두 DB가 담당

---

## 2. 예상되는 구조적 한계 및 문제점

해당 구조는 초기에는 단순하고 안정적이지만, 데이터 증가에 따라 다음과 같은 한계를 갖는다:

### 2.1 작업 집중으로 인한 자원 병목

- 데이터는 유입량은 유동적이지만, 처리는 특정 시점에 고정되어 시스템 부하 급증 가능성

### 2.2 반영 지연 및 사용자 경험 저하

- 분석 결과의 최신성이 고정된 배치 주기에 종속됨
    - 배치 주기 단축 시, 변경이 없는 음식점까지 반복 분석되어 비용 증가
    - 배치 주기 증가 시, 과부하 및 분석 결과 반영 지연 발생
    - 즉, 단일 배치 주기로는 비용 효율성과 데이터 최신성을 동시에 만족하기 어려움
- 신규 리뷰 개수 기반 필터링 방식은 불완전함
    - 일부 최적화는 가능하나, 시간 기반 실행 구조는 유지됨
    - 여전히 음식점 별 트래픽 편차를 반영하지 못해 과부하 또는 지연 발생

### 2.3 RDB 과부하 및 오케스트레이션 비용 증가

- 음식점 단위 작업의 상태 전이를 DB로 관리
- 작업 실행을 위한 polling 조회 및 상태 갱신(write)이 반복적으로 발생
- 이러한 오케스트레이션 I/O가 실제 분석 작업과 함께 수행되며 DB 부하를 추가적으로 증가시킴

### 2.4 결론

- **정적 스케줄링으로는 데이터 유입 패턴을 반영할 수 없으므로, 유동적 처리 구조가 필요함**

---

## 3. 부하 테스트를 통한 문제 검증

부하 테스트 결과, 다음과 같은 패턴이 확인되었다:

- 배치 시작 직후 `vector_upload` 작업이 한꺼번에 많이 생성됐다.
- executor active thread 는 빠르게 최대치 8개에 도달했다.
- queue는 빠르게 1000 가까이 증가했다.
- 실제 `RejectedExecutionException`이 발생했다.

이는 배치 시작 시점에 작업이 집중 생성되면서 처리 속도를 초과하는 구조적 한계를 드러낸다. 단순한 스레드 수 조정이나 서버 증설로 해결할 수 있는 문제가 아니라, 작업 생성 시점의 집중을 완화하는 구조적 개선이 필요함을 확인하였다.

---

## 4. 고도화 필요성 및 목표

위 문제를 해결하기 위해 다음과 같은 방향의 구조 개선이 필요하다:

- **작업 몰림 완화**: 시간 기반으로 집중 생성되던 작업을 이벤트 기반으로 분산 처리
- **처리 안정성 확보**: 작업 생성 속도와 처리 속도 간 불일치를 허용하고, queue 기반으로 완충하여 과부하 상황에서도 시스템이 실패하지 않도록 개선
- **처리 지연 감소**: 배치 의존 구조를 탈피하여 데이터 변경 시점에 가까운 시점에 분석 수행
- **처리량 확장 가능 구조로 전환**: consumer 수 증가를 통해 처리량을 선형적으로 확장할 수 있는 구조로 개선

## 5. 아키텍처 전환 방향

### 5.1 분석 단위

리뷰 분석은 **개별 리뷰 단위가 아닌 음식점 단위 전체 상태를 재계산하는 구조**를 갖는다. 따라서 리뷰 생성 이벤트를 기반으로 하되, 이벤트 발생 시점마다 즉시 처리하는 방식은 중복 계산을 유발하므로 적합하지 않다.

### 5.2 처리 방식

리뷰 분석 파이프라인은 각 단계의 특성에 따라 서로 다른 처리 방식을 조합하여 구성한다.

| 단계 | 처리 방식 | 이유 |
| --- | --- | --- |
| 벡터 업로드 | Micro-batching | 리뷰 집합을 기반으로 상태를 집계하는 작업 |
| 리뷰 분석 | Stream (event-triggered) | 이미 집계된 상태를 기반으로 실행 |

### 5.2.1. Micro-batching 적용 근거

리뷰 분석은 음식점 단위 전체 상태를 재계산하는 작업으로, 개별 이벤트마다 실행할 경우 동일한 데이터에 대한 중복 계산이 반복된다.

따라서 리뷰 이벤트를 일정 시간 동안 집계한 뒤 한 번에 처리하는 micro-batching 방식이 비용 효율 측면에서 적합하다.

또한 이벤트가 지속적으로 유입되는 경우에도 무한정 지연되지 않도록 Idle time과 Max delay를 함께 적용하여 처리 시점을 제어한다.

### 5.2.2. 리뷰 분석 단계의 Stream 처리 근거

벡터 업로드 단계에서 이미 음식점 단위로 데이터가 집계되므로, `vector_uploaded` 이벤트는 해당 시점의 최신 상태를 반영한 완결된 입력으로 볼 수 있다.

따라서 리뷰 분석 단계에서는 추가적인 batching을 수행하지 않고, 해당 이벤트를 즉시 처리하는 방식이 더 적합하다.

### 5.3 처리 구조

```
리뷰 이벤트
→ 음식점 단위 이벤트 집계 (Micro-batching)
→ 벡터 업로드
→ vector_uploaded 이벤트 생성 (음식점당 1건)
→ 리뷰 분석 (즉시 실행)
```

### 5.4 설계 원칙

### 5.4.1. 이벤트 밀도 기반 Adaptive 처리 (Latency–비용 균형)

리뷰 이벤트의 유입 패턴은 음식점마다 크게 다르므로, 고정된 배치 주기로는 효율적인 처리가 어렵다.

이를 해결하기 위해 **Idle time과 Max delay를 함께 사용하는 Adaptive micro-batching 전략**을 적용한다.

- 이벤트 많음 → Idle time이 충족되지 않아 자연스럽게 batching
- 이벤트 적음 → Max delay에 의해 빠르게 실행
- Idle time: batching 효율 확보 (비용 감소)
- Max delay: 지연 상한 보장 (Latency 확보)

→ **Hot key / Cold key 모두 대응하면서, 비용과 응답성을 동시에 만족**

### 5.4.2. 처리 안정성 확보

- 메시지 큐 기반 비동기 처리
- queue를 통한 buffer 확보 (backpressure 대응)

→ **과부하 상황에서도 실패가 아닌 지연으로 전환**

### 5.4.3. 장애 복구 및 데이터 정합성 확보

- DB 기반 상태 관리 (`lastEventTime`, `lastAnalyzedAt`)
- At-least-once + idempotent 구조

→ **이벤트 유실 없이 자동 복구 가능**

### 5.5 예상 가능한 리스크

### 1) Batch 크기 및 실행 시점 제어

Micro-batching은 실행 시점을 단일 조건으로 결정하지 않고, 다음 세 가지 조건을 조합하여 제어한다:

- Idle time: 이벤트가 멈춘 시점 감지
- Max delay: 최대 대기 시간 제한
- Max items: batch 크기 상한 제한

| 조건 | 목적 | 잘못 설정 시 문제 |
| --- | --- | --- |
| Idle time | 이벤트 종료 감지 | 너무 작으면 batching 감소, 너무 크면 지연 증가 |
| Max delay | 무한 대기 방지 | 너무 작으면 비용 증가, 너무 크면 지연 발생 |
| Max items | batch 크기 제한 | 너무 작으면 비효율, 너무 크면 처리 시간 증가 및 payload 문제 |

# 6. 재시도 처리 전략

## 6.1 문제 정의

Micro-batching 구조에서는 하나의 작업이 여러 리뷰 이벤트를 포함하므로, 장애 발생 시 다음 문제가 발생한다:

- 어떤 리뷰까지 처리되었는지 불명확
- 동일 batch 재실행 필요
- Kafka 재전달만으로는 동일 입력 보장 어려움
- 재시도 시 실행 순서 역전(out-of-order) 가능성 존재

---

## 6.2 설계 원칙

재시도는 단순 메시지 재수신이 아니라 다음을 만족해야 한다:

**동일 입력에 대해 동일 결과를 생성하고, 순서 역전에도 상태가 깨지지 않아야 한다**

---

## 6.3 해결 방식

### 1) 이벤트 자체가 batch 경계를 정의

```json
{
  "restaurant_id": 1,
  "review_ids": [A, B, C]
}
```

- `review_ids` 자체가 batch의 입력을 명확히 정의
- 재시도 시 동일 메시지를 그대로 재처리 가능

→ **입력 불변성 확보**

---

### 2) At-least-once 기반 재처리 허용

- Kafka 특성상 동일 이벤트 중복 전달 가능
- 이를 전제로 설계

→ **중복 실행은 정상 흐름으로 간주**

---

### 3) Snapshot overwrite + Epoch 기반 멱등성

리뷰 요약 및 감정 분석 결과는 누적이 아닌 snapshot으로 저장한다.

```
restaurant_id 단위 상태
- summary
- sentiment
- epoch
```

처리 로직:

```
if (event.epoch <= current_epoch)
    → discard

→ 분석 수행
→ 결과 overwrite + epoch 갱신
```

- 최신 결과만 유지
- 재시도 / 중복 / 순서 꼬임 자동 방어

→ **idempotency + ordering 문제 동시 해결**

---

## 6.4 한계 및 고려사항

- batch 크기 증가 시 메시지 크기 제한 가능성
- late event는 다음 batch에서 반영됨 (완전 재현 불가)
- epoch 관리 필요 (restaurant 단위 상태)

---

## 6.5 결론

재시도는 메시지 큐가 아니라
**입력 불변성 + snapshot overwrite + epoch 기반 멱등성으로 해결한다**

---

# 7. Window 기반 Micro-batching 설계

## 7.1 개요

리뷰 이벤트를 음식점 단위로 집계하여 **리뷰 요약 및 감정 분석을 batch 단위로 수행**하기 위해 key 기반 windowing을 적용한다.

---

## 7.2 상태 구조

```
Map<restaurantId, WindowState>
```

```
WindowState
- reviewIds (buffer)
- size
- windowStart
- lastEventTime
- state (IDLE / COLLECTING / FLUSHING)
```

---

## 7.3 실행 조건

다음 조건 중 하나를 만족하면 batch 실행:

- **Max items**: 일정 개수 이상 수집
- **Idle time**: 마지막 이벤트 이후 일정 시간 경과
- **Max delay**: 첫 이벤트 이후 최대 대기 시간 초과

→ **시간 + 크기 기반 hybrid trigger**

---

## 7.4 상태 전이

```
IDLE → COLLECTING → FLUSHING → IDLE
```

- IDLE: 이벤트 없음
- COLLECTING: 리뷰 수집 중
- FLUSHING: 분석 요청 수행 중

---

## 7.5 Window 분리 (Double Buffering)

Flush 중 새로운 이벤트가 들어오면:

- 기존 window → FLUSHING 유지
- 새로운 window 생성 → COLLECTING 시작

→ **batch 간 데이터 혼합 방지 및 race condition 제거**

---

## 7.6 설계 효과

- 이벤트 밀도에 따른 adaptive batching
- 불필요한 분석 요청 감소
- 처리량 및 비용 효율 개선

---

## 7.7 한계 및 트레이드오프

### 1) 파라미터 튜닝 문제

| 항목 | 너무 작을 경우 | 너무 클 경우 |
| --- | --- | --- |
| Idle time | batching 효과 감소 → 비용 증가 | hot key에서 flush 지연 |
| Max delay | 빠른 응답 but 비효율 | cold key에서 반영 지연 |
| Max items | 작은 batch → 비효율 | 큰 batch → 지연 증가 |

→ **Throughput vs Latency 트레이드오프**

---

### 2) 상태 유실 가능성

- in-memory window는 장애 시 유실됨
- offset 재처리와 결합 시 batch 재구성 가능

→ epoch 기반 방어 필요

---

### 3) 복잡도 증가

- 상태 관리
- window lifecycle 관리
- double buffering

---

## 7.8 결론

Windowing은 단순 이벤트 처리 방식이 아니라 **batch 경계를 정의하는 상태 기반 제어 메커니즘이다**

---

# 8. Kafka Streams 미사용 이유

## 8.1 대안

Kafka Streams 대신, 일반 Kafka consumer + 애플리케이션 로직으로 windowing을 직접 구현하였다.

---

## 8.2 Kafka Streams의 장점

- windowing 및 state 관리 내장
- 장애 복구 및 state restore 지원
- stream processing abstraction 제공

---

## 8.3 미사용 이유

### 1) 도메인 특화 처리

- 음식점 단위 aggregation
- Idle time + Max delay + Max items 조합
- AI 기반 리뷰 요약 / 감정 분석 호출

→ 일반적인 time-window 모델로 표현 어려움

---

### 2) 복잡도 대비 효용 부족

- 높은 러닝 커브
- 디버깅 어려움
- state store 및 rebalance 관리 필요

→ 현재 요구사항 대비 과도한 복잡도

---

### 3) 서비스 로직 중심 구조

- 데이터 플랫폼이 아닌 **백엔드 기능 구현**
- 특정 파이프라인에 국한된 처리

→ 범용 스트림 엔진 도입 필요성 낮음

---

## 8.4 선택 근거

본 시스템은 범용 스트림 처리가 아니라 **도메인 중심 batch 처리이므로 직접 구현이 더 단순하고 명확하다**

---

## 8.5 향후 확장 가능성

다음 경우 Kafka Streams 도입 고려:

- 다수 aggregation 로직 등장
- windowing 재사용 필요
- 상태 복구 및 운영 복잡도 증가

---

## 8.6 결론

Kafka Streams는 강력하지만, 현재 시스템에서는 **직접 구현이 더 단순하고 적합한 선택이다**
